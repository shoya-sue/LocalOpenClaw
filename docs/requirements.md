## 1. プロジェクト概要

### ゴール

ローカルPC上で完全自律動作するAIエージェントチームを、誰でも簡単に起動・運用できるようにする。

### コンセプト

- **完全ローカル**: クラウド依存ゼロ。APIキー不要。
- **人格継続**: 再ビルド後も記憶・人格が引き継がれる。エージェントは「育つ」存在。
- **役割分担チーム**: 探偵・研究者・営業・秘書・エンジニアが連携して動く。
- **視認性**: ピクセルアート（ドット絵）UIでチームの状態をリアルタイム可視化。
- **データ主権**: AIに渡すデータは事前スクリプトで厳選。AIは「判断」のみ担当。
- **段階的データ拡張**: 小さなデータセットから始め、精度を確認しながら範囲を広げる。

## 3. エージェント設計

### 3.1 人格継続の仕組み

```
再ビルド（docker-compose down → up）しても「別人格」にはならない理由:

┌──────────────────────────────────────────────────────┐
│  エージェントの「アイデンティティ」= 2ファイルで構成     │
│                                                      │
│  1. キャラクター定義ファイル（YAML/TOML）              │
│     └── 人格・役割・口調・行動原則・専門知識            │
│         Git管理 → バージョン管理・引き継ぎ可能          │
│                                                      │
│  2. Markdownメモリファイル（OpenClaw永続メモリ）       │
│     └── 経験・学習した知識・人間関係・過去の会話        │
│         Dockerボリュームマウント → 再ビルド後も残る     │
└──────────────────────────────────────────────────────┘
```

**設定例（キャラクター定義）:**
```yaml
# config/agents/detective.yaml
name: "探偵"
codename: "detective"
personality: |
  あなたは情報収集を専門とする探偵です。
  好奇心旺盛で、小さな手がかりから全体像を掴む能力があります。
  収集した情報は必ず裏付けを取り、研究者に正確に引き渡します。
role: "information_gathering"
tools:
  - web_search
  - file_read
  - database_query
memory_path: "/workspace/detective/memory/"
```

### 3.2 エージェントチーム構成

最小構成（Phase 1）と完全構成（Phase 4）の2段階で実装する。

#### 完全構成（5エージェント）

| エージェント | 役割 | 主なツール | 性格イメージ |
|------------|------|----------|------------|
| **リーダー** | チーム指揮・最終判断・タスク割り振り | 全エージェントへの指示・状態監視 | 冷静・決断力・全体俯瞰 |
| **探偵** | 情報収集・調査・手がかり発見 | Web検索・ファイル読み込み・DB検索 | 好奇心旺盛・執念深い・細部注目 |
| **研究者** | 収集情報の検証・分析・仮説立証 | 論文参照・比較分析・スコアリング | 論理的・懐疑的・根拠重視 |
| **営業** | 情報の運搬・整理・フォーマット化 | データ変換・ファイル書き出し・配信 | 整理整頓・実行力・橋渡し役 |
| **秘書** | スケジュール管理・タスク優先度管理 | カレンダー操作・リマインダー・ログ | 几帳面・先読み・サポート気質 |
| **エンジニア（土方）** | システム構築・コード実装・環境整備 | コード生成・テスト実行・デプロイ | 職人気質・実装優先・黙々と作業 |

#### 最小構成（Phase 1 MVP: 3エージェント）

```
リーダー ─── 探偵（情報収集）
    └──── エンジニア（実行・構築）
```

### 3.3 エージェント間の連携フロー

```
[ユーザー] → 「FPSゲームを作りたい」
    ↓
[リーダー] タスク分解・割り振り
    ↓
[探偵] 銃の歴史・戦争・部隊編成などを収集
    ↓
[営業] 収集データを整理・データパイプラインへ投入
    ↓
[研究者] データの正確性検証・ゲーム設計への適用可否チェック
    ↓
[秘書] 実装タスクのスケジュール化・優先度整理
    ↓
[エンジニア] 実際のコード実装・テスト
    ↓
[リーダー] 成果物確認・ユーザーへ報告
```

| **UIゲームエンジン** | Phaser.js | Star-Office-UIで実証済み・スプライト対応 |
| **UIコンポーネント** | Pixelact UI (React + shadcn) | ピクセルアートデザインシステム・MIT |
| **バックエンドサーバー** | Python + FastAPI | 非同期・WebSocket対応 |
| **コンテナ** | Docker + Docker Compose | OpenClaw公式構成・ボリューム永続化 |

| 項目 | 最小 | 推奨 |
|------|------|------|
| RAM | 8GB（7Bモデル） | 16-32GB（13B+モデル・マルチエージェント） |
| ストレージ | 40GB | 100GB+ |
| GPU | Apple Silicon Metal | M2/M3 Pro+（統合メモリ16GB以上） |

## 6. データパイプライン設計

### 設計原則

> AIは「判断」のみ。データ投入前にスクリプトで厳選する。
> 小さなデータセットから始め、精度を確認しながら範囲を広げていく。

### データ投入の段階的拡張（ユーザー定義）

```
Phase A: 最小データセット（動作確認）
  └── 手動で選んだ5〜10件のテキストファイル

Phase B: テーマ別データセット
  └── 例: FPSゲーム制作 → 銃の歴史・戦争・部隊編成
         ↓ スクリプトで関連度スコアリング
         ↓ 高スコアのみChromaDBへ格納
         ↓ AIは格納されたもののみを参照

Phase C: 全データソース統合
  └── ローカルファイル / Git / ブックマーク / ログ / カレンダー 全て
         ↓ 自動フィルタリング + 重複除去
         ↓ 定期再インデックス（cron）
```

### 対応データソース（全て対応）

| データソース | 取り込み方法 | フィルタリング例 |
|------------|------------|--------------|
| ローカルファイル | watchdog でファイル監視 | 更新日時・ファイルサイズ・キーワード |
| Gitリポジトリ | git log / git diff 解析 | コミットメッセージ・変更ファイル数 |
| ブックマーク | ブラウザエクスポート → パース | ドメイン・タグ・最終アクセス日 |
| Web記事 | URLリスト → スクレイピング | 文字数・品質スコア・重複率 |
| カレンダー | iCal / Google Calendar API | 日付範囲・カテゴリ |
| タスクリスト | Markdown / JSONエクスポート | 優先度・期限・ステータス |

### 前処理パイプライン

```
[生データ]
  ↓ unstructured（PDF/HTML/Word等のパース）
[テキスト抽出]
  ↓ セマンティックチャンキング
[チャンク化]
  ↓ 品質スコアリング（Nemotron式5軸評価）
  ↓ 重複排除（ハッシュ + セマンティック）
[フィルタリング済みデータ]
  ↓ Ollama embedding（nomic-embed-text等）
[ベクトル化]
  ↓ ChromaDB / Qdrant
[検索可能な知識ベース]
  ↓ RAG（LlamaIndex）
[AIへのコンテキスト投入]
```

## 7. UIデザイン方針

### ピクセルアートオフィス（エージェント可視化）

Star-Office-UI（MIT）をベースにフォーク。6エージェント対応にカスタマイズ。

#### エージェント状態とビジュアル

| 状態 | ビジュアル |
|------|----------|
| `working` | デスクでタイピングアニメーション |
| `thinking` | 吹き出しに「...」 + 思考内容プレビュー |
| `communicating` | エージェント間に矢印エフェクト |
| `idle` | 休息エリアでアイドルアニメーション |
| `error` | バグゾーン移動 + 赤エフェクト |

#### オフィスゾーン構成

```
┌──────────────────────────────────────────────┐
│  調査エリア       │  分析エリア               │
│  [探偵]          │  [研究者]                 │
│  情報収集中...    │  データ検証中...           │
├──────────────────┼──────────────────────────┤
│  リーダー室      │  作業エリア               │
│  [リーダー]      │  [エンジニア(土方)]        │
│  チーム指揮中... │  コーディング中...         │
├──────────────────┼──────────────────────────┤
│  物流エリア      │  スケジュール室            │
│  [営業]          │  [秘書]                   │
│  データ整理中... │  タスク管理中...           │
└──────────────────────────────────────────────┘
┌──────────────────────────────────────────────┐
│  操作パネル: ログ | タスクキュー | データ投入 │
└──────────────────────────────────────────────┘
```

## 10. リスクと課題

| リスク | 影響度 | 対策 |
|--------|--------|------|
| ローカルモデルの精度（多エージェント時） | 高 | モデル選定検証・エージェントごとにモデルサイズを変える |
| RAM不足（6エージェント同時動作） | 高 | エージェントを順次起動・軽量モデル採用・量子化（Q4_K_M） |
| アート素材のライセンス | 中 | Phase 3でオリジナル素材作成 |
| エージェント間の役割衝突 | 中 | リーダーによる明確なタスク割り振りルール定義 |
| OpenClawのバージョン変化 | 低 | バージョンロック管理 |
